Flutter Project Architecture

Three ways to structure the Flovi Driver app. All use Riverpod + go_router.

A

Feature-First (Recommended)

Each feature is a self-contained folder with its own screens, providers, and models. No domain/data layer separation — just clean, practical Flutter.

lib/
├── core/
│   ├── api/         # HTTP client, endpoints
│   ├── auth/        # Supabase auth service
│   └── router/      # go_router setup
├── features/
│   ├── auth/        # login screen, auth provider
│   ├── available_jobs/  # browse pending
│   ├── my_jobs/         # booked/in-progress
│   └── profile/
└── shared/
    └── widgets/     # reusable UI

Pros

  • Easy to navigate
  • Feature agents work independently
  • No over-engineering

Cons

  • Less strict separation
B

Clean Architecture

Strict presentation / domain / data layer separation per feature. Repositories, use cases, entities. Maximum testability and structure.

lib/features/available_jobs/
├── presentation/
│   ├── screens/
│   └── providers/
├── domain/
│   ├── entities/
│   └── repositories/  # abstract
└── data/
    ├── models/
    └── repositories/  # concrete

Pros

  • Very testable
  • Enterprise-grade

Cons

  • High boilerplate for 4 screens
  • Slower to build
C

Flat / Type-First

Organized by type — all screens together, all providers together, all models together. Simple, fast to scaffold.

lib/
├── screens/
│   ├── login_screen.dart
│   ├── available_jobs_screen.dart
│   └── my_jobs_screen.dart
├── providers/
├── models/
└── services/

Pros

  • Minimal structure
  • Fastest to start

Cons

  • Doesn't scale
  • Hard to assign to teams

My recommendation: A — Right balance of structure and speed. Feature folders map cleanly to teammate assignments.
