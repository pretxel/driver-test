Data Models, API Layer & State

Core Data Model

// lib/core/models/relocation.dart
class Relocation {
  final String id;
  final String vehicleMake;
  final String vehicleModel;
  final String pickupLocation;
  final String dropoffLocation;
  final RelocationStatus status;  // enum
  final DateTime createdAt;
}

enum RelocationStatus {
  pending,     // Available to book
  inProgress,  // Driver assigned
  completed,
  cancelled,
}

API Layer (Dio + Supabase JWT)

Base URL: https://vfmtrozkajbwaxdgdmys.supabase.co/functions/v1/api

GET  /api/v1/relocations          → List<Relocation>  (my jobs)
POST /api/v1/relocations          → List<Relocation>  (pending list)
PUT  /api/v1/relocations/{id}     → Relocation        (book job)

All requests include:
  Authorization: Bearer <supabase_access_token>
  Content-Type: application/json

A Dio interceptor reads the token from Supabase's active session and injects it automatically — no manual token handling per request.

Riverpod State Providers

Provider Type Owns
authStateProvider StreamProvider Supabase auth session stream
availableJobsProvider AsyncNotifierProvider Pending relocations list + refresh
bookJobProvider AsyncNotifierProvider PUT booking call + optimistic update
myJobsProvider AsyncNotifierProvider IN_PROGRESS / COMPLETED / CANCELLED list

Confirmation Flow (One-Tap Booking)

1. Driver taps a pending job card
2. Bottom sheet opens showing job details + CONFIRM BOOKING button
3. Tap confirm → loading state → PUT /api/v1/relocations/{id}
4. On success → job removed from available list, appears in My Jobs as IN_PROGRESS
5. On error → snackbar with error message, state rolled back
